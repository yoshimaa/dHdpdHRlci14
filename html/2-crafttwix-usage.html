Craft Twixの使い方(予約投稿編)

Googleスプレッドシートの新規作成

次のURLにアクセスしてコピーを作成します。作成済の方は必要ありません。

Googleスプレッドシートのテンプレート

ファイル > コピーを作成

使用する前にAPIの設定が必要です。APIをまだ持っていない方は、API取得に関する説明欄をご参照ください。

Googleスプレッドシートのコマンド取り扱い説明書

画面右上に表示されるカスタムメニューに次のコマンドがあります。

  • X(Twitter)の予約投稿始動!
  • X(Twitter)の予約投稿終了!
  • WordPressのカテゴリを登録する
  • WordPressの記事を定期投稿する
  • WordPressの記事を定期投稿しない
  • X(Twitter)のAPIを登録する!
  • X(Twitter) Access Tokenを登録する!
  • X(Twitter) APIの認証!
  • WordPress APIを登録する!
  • Googleドライブの保存先
  • Googleドライブに画像をアップロードする
  • GoogleドライブのビデオURLを指定する
  • 日付の一括変換

ここではメニューを一通り説明させてもらいます。WordPressのコマンドは簡易的な説明にと留めています。詳細はWordPress編をみてください。

● X(Twitter)の予約投稿始動!

X(Twitter)の予約投稿を開始します。APIの登録と認証をあらかじめ済ませる必要があります。1時間おきに1時間以内の投稿が15個まで登録されます。

また、以下のものも登録されません。

  • X(Twitter) API Freeは1日50投稿、月1500が上限になっています。
  • X(Twitter)に負荷をかけないよう、同じ時間の投稿は登録されません。最低1分以上ずらしてください。
  • 登録OK？が完成になっていないものは登録されません。下書きをうっかりと登録しないようになっていますので安心して書けます。
  • 完成になっていても、日付、時間、テキストが1文字も記入されていない場合は登録されません。
  • 終了している投稿も登録されません。終了日を確認してください。


● X(Twitter)の予約投稿終了!

X(Twitter)の予約投稿を終了します。APIの登録と認証をあらかじめ済ませる必要があります。

  • 終了すると、予約された投稿がGAS上から削除されます。
  • ユーザーが入力したスプレッドシート上には影響がないためご安心ください。ただし、実施日などプログラミングから書き込まれた箇所は一度クリアされます。


● WordPressのカテゴリを登録する

Webサイトのカテゴリを登録できます。カテゴリを登録すると、カテゴリごとにハッシュタグを割り当てることができます。

● WordPressの記事を定期投稿する

WordPressの記事を定期投稿します。タイトルとURL、カテゴリごとにタグを設定できます。

投稿時にハッシュタグをつけたい場合は、カテゴリの登録とタグの登録が必要です。

● WordPressの記事を定期投稿しない

WordPressの記事を定期投稿しません。シートは一度クリアされます。

● X(Twitter) APIを登録する!

X(Twitter)のAPIを登録します。詳しくは後述しています。

● X(Twitter) Access Tokenを登録する!

X(Twitter)のAccess Tokenを登録します。詳しくは後述しています。

Access TokenはCallbackURLなどを変更すると、再登録が必要です。


● X(Twitter) APIの認証!

X(Twitter)の認証をします。詳しくは後述しています。

● WordPress APIを登録する!

WordPress APIを登録します。詳しくはWordPressの説明欄をみてください。

● Googleドライブの保存先

Googleドライブの保存先を設定します。「画像・動画編」で詳しく説明しています。

● Googleドライブでの画像アップロード

Googleドライブの画像アップロードは、プレビュー画面を見ながら画像の並べ替えが可能です。「画像・動画編」で詳しく説明しています。

● GoogleドライブのビデオURL指定

動画はGoogleドライブの通常のアップロード手順で行い、URLのみを指定します。「画像・動画編」で詳しく説明しています。

● 日付の一括変換

スケジュール投稿の日付を一括でずらすことができます。詳細は「日付の一括変換」の項目に記載しています。

権限の許可

カスタムメニューから何らかのメニューを実行しようとすると、権限の許可が求められます。

カスタムメニュー > XXX

Google Apps Script (GAS)をはじめて動かす前の承認作業になります。この作業はスプレッドシートとGASを使っていると、定番の作業なので覚えておきましょう。

権限を許可するための手順は次のとおりです。

続行 > 該当メールアドレスでログイン > 詳細 > プロジェクト名(安全ではないページ)に移動 > 許可

続行をおして該当メールアドレスでログインします。

詳細をおしてプロジェクト名(安全ではないページ)に移動します。

安全ではないページとでますが、プログラミングで外部とやりとりする場合、このような警告が表示されることはよくあります。これはGASが外部のサービスと通信するための許可を求めているためです。心配しなくてよいでしょう。許可しないと利用できません。

最後に、許可のボタンをおすと承認が成功します。スクリプトが実行できるようになります。

スプレッドシートへのAPI情報の登録

先にメモしておいたAPIキーをスプレッドシートに入力します。APIキーが足りない場合は、前の項目を再確認してください。

半角スペースが入っている(こちらは自動削除されるようにしています)、または最後の1文字が選択されていないなど、APIキーが正しく入力されていないと、スクリプトが正常に動作しません。もし何か問題があった場合は、APIキーを再入力しましょう。

カスタムメニュー > X(Twitter) APIを登録する!

  • Consumer API Key(初回はアプリ作成時に取得)
  • Consumer API Secret(初回はアプリ作成時に取得)
  • Client ID(User authentication settingsで取得)
  • Client Secret(初回はUser authentication settingsで取得)
  • Bearer Token(初回はアプリ作成時に取得)

各APIキーを入力して保存ボタンを押します。

カスタムメニュー > X(Twitter) Access Tokenを登録する!

  • Access Token(Keys and tokensで取得)
  • Access Token Secret(Keys and tokensで取得)

各APIキーを入力して保存ボタンを押します。

注意：もしUser authentication settingsの設定を変更した場合は、Access TokenとAccess Token Secretを再生成し、再度入力する必要があります。そのため、アクセストークンのメニューは別になっています。

スプレッドシートからX(Twitter)に認証

まだあるのかと思われる方もいるかもしれませんが、もう少しです。

APIキーを入力したのち、認証を行います。

カスタムメニュー > X(Twitter) APIの認証!

URLが表示されますので、それをクリックします。成功すると、X(Twitter)にアプリケーションが追加されます。

この認証時によくでるエラーを紹介します。

error

アプリにアクセスを許可できません。 前に戻ってもう一度ログインしてください

このエラーは、Client IDやClient Secretのいずれか、あるいは両方が間違っている可能性があります。これらを修正することでエラーは解消されます。

別のパターンとして、指定したコールバックURLが入力されていない、または正しくない場合にも発生します。コールバックURLを変更すると、Access TokenとAccess Token Secretを再生成して更新する必要があります。

お疲れ様でした。ここまでくると、いよいよCraft Twixが使えます。

スケジュール投稿の項目取り扱い説明書

ここまでくると、予約投稿の方法は簡単です。
スプレッドシートに情報を入力した後、X(Twitter)の予約投稿始動!を実行するだけです。

ただし、最初は1行や数行だけでテストを行い、動作確認をしっかりと取ることをおすすめします。シートの使い方を自分の中できちんと把握することが大事だからです。その確認作業が終わった後に、大量に予約投稿を用意しましょう。

シート名はschedule-tweetです。シート名はプログラミングから参照しているため変更しないでください。

各々の背景の色には次の意味があります。

  • ブルー：必須項目です。
  • グリーン：任意項目です。
  • バイオレット：プログラミングから書き込みます。ただしユーザーが書き込んだ場合、その値がプログラミングで書き込む値より大きい場合は優先されます。平たく言うと、終了日を長めに設定することができます。終了日の設定のみに使われています。
  • グレー：プログラミングから書き込みます。指定フォーマット以外の値が入ると問題が発生するため、基本的にいじらないでください。ただし、何らかの問題が発生してリセットしたい場合は、入力内容を消すのはOKです。
  • ピンク：文字数オーバーするとピンクのセルになります。

各シートの説明をします。

● NO

通しNOです。1、2、3と番号を振ってください。

● 次回の実施日

  • プログラミングから書き込まれるため、ユーザーが設定する必要がありません。
  • 新規でスケジュール登録するとき、投稿が完了したあとに値が更新されます。
  • カスタムメニューからスケジュールを開始したとき、実施日のシートは再生成されます。
  • カスタムメニューからスケジュールを停止したとき、実施日のシートはクリアされます。

必須項目が入力されていないと、次回実施日が登録されません。登録OK?が完成になっている必要があります。

● 年月日

初回実施の年月日です。フォーマットは2023/00/00です。フォーマットは原則変えないでください。

● 時間

時間です。昼の時間は背景が自動的に黄色になります。フォーマットは00:00です。フォーマットは原則変えないでください。

● インターバル

  • 背景がグリーンなので、入力するかどうかは任意です。
  • 何日ごとに投稿するのかを決めます。重複投稿にならないよう最低24時間の間隔があく日にち単位で設定できます。基本的に1でも問題ありませんが、短い日にちを設定するとうっすらと背景色が赤色になります。余裕を持って自分で管理してください。
  • 昔から時間をあけると重複投稿のチェックにはかかりません。ただ、重複投稿を1つも作らないよう、サブテキストのシステムを用意しています。より安全に運用できます。
  • サブテキストを使うと、いくつかのバリエーションを持たせて投稿することができます。そのため、重複投稿はできません。詳細はサブテキストの欄を見てください。
  • インターバルは任意項目なので使いこなせる人だけ使ってください。難しい人のために、初心者向けの機能として、カスタムメニューから日付を一括でずらす便利な機能を付けています。

● 終了日

  • 終了日はプログラミングから自動で設定されます。
  • インターバルの指定がない場合は、指定した年月日と時間と一緒になります。1回のみ投稿されておしまいです。
  • インターバルの指定がある場合は、インターバルに3をかけた日付に設定されます。重複投稿をさけるため、この機能はサブテキストの利用を前提としています。サブテキストは3パターン用意されているため、サブテキストを使って重複しない投稿が実現できます。たとえばインターバルを30日にすると30日ごとに投稿されます。90日目に3パターンの投稿を行って終了します。
  • なお、中級者向けに裏技があります…。終了日はプログラミングから自動で設定されますが、ご自身でそれより長い期間を入力して決めることが可能です。その場合はループの挙動になります。
  • この方法で運用する場合は、投稿を削除するサービス等とあわせて使うと重複投稿は確実に避けられます。
  • 1時間ごとなど短時間での同じ投稿は重複投稿となります。ただ、期間をあけた場合、WordPressの過去投稿を繰り返し投稿するプラグインや以前のTwitter予約投稿ツールなどでも今まで現実的に問題になったことはありません。しかし、重複投稿の期間は明言されておらず基本的に望ましくないとされているため、サブテキストシステムや投稿を削除するサービスとあわせて使うことにより、重複投稿はさけつつ安全に運用することをおすすめします。
  • 本当はX(Twitter) APIの自動リツイートで実装する方が望ましかったのですが、X(Twitter) APIの無料版では使えないためこのような仕様となりました。
  • 終了日は多忙な方に便利な機能かつ要望もわりと多いことを知っているため残しましたが、短期間のインターバルは重複投稿になる可能性がありますので十分にご注意の上、ご利用ください。重複投稿の凍結は今まで個人的に発生したことがありませんが、こちらでは凍結等の責任は持てません。心配な方は利用しないことをおすすめします。ただ、サブテキストなどを利用するなど安全に運用できるよう、できる限り配慮はしました。

● 登録OK?

  • 「作成中」と「完成」を選べます。未入力は「作成中」とみなされます。
  • 「完成」にしないとスケジュール投稿されません。
  • 作業中の項目が勝手に登録されて投稿されないため、安心感があります。
  • 書きかけの項目は「作成中」にしておきましょう。
  • 念のため補足しておくと、セレクトボックスも一括でコピペできます。1つ1つ選ぶ必要はありません。

● ツリー投稿

  • 「通常投稿」と「ツリー投稿」を選べます。未入力は「通常投稿」とみなされます。
  • ツリー投稿はわかりやすく、上の行にスレッドとして投稿されます。
  • 時間差をつけることが可能です。
  • 関連づける投稿がないと投稿されませんので注意してください。
  • 前の投稿より必ず未来の時間に設定してください。

ここまでで半分弱ぐらいシートの解説をしました。
試すのが早いのですが、頭を整理するためにも一度動画を見てみましょう。慣れれば簡単です。

動画の解説をします。

1行目と2行目はインターバルが指定されています。
そのため、インターバルにサブテキストの3回文をかけた日付が終了日として設定されました。

3行目は実施日も終了日も入力されませんでした。何故かといえば「完成」には変更されていますが、年月日と時間が指定されていないため、登録外と判定されています。このように必須項目が抜けているものは登録されません。

4行目と5行目はツリー投稿です。インターバルが指定されていないため実施日と終了日ともに同じ日付が入力されています。予約投稿は1回行って終了です。5行目は4行目の15分後という未来の時間に指定されています。

終了日は基本的にプログラミングが設定しますが、裏テクで自分で後から変更することも可能です。

もし誤って予約投稿を開始してしまった場合、すぐに停止できます。その際、実施日は削除され、GASの予約投稿もクリアされます。ただし、終了日はユーザーが操作できる仕様なので、自動で削除されません。必要に応じて自分で削除してあげましょう。

シートの解説を続けます。

● メインテキスト

  • 投稿の本文です。
  • 日本語は140文字までです。
  • X(Twitter) Blueの文字数にはAPIの仕様上、対応できないためご容赦ください。
  • テキストを入力すると、その右隣の文字数に反映されます。

● 文字数

  • 左のセルの文字数をカウントしています。URLも改行もX(Twitter)の仕様にあわせてカウントされます。
  • 改行は0.5文字。
  • URLは一律23文字。
  • 顔文字は特殊のため、ゆとりをもってカウントされるようになっています。若干少なくカウントされることがありますが、140文字以内におさまればオーバーすることはありません。
  • テキストをコピペで移植したとき、スプレッドシートの仕様上、きちんと文字数がカウントされない場合があるため、再度メインテキストやサブテキストの欄でEnterをおしてください。

● サブテキスト1

  • サブテキストの利用は任意です。サブテキストの列は、列を閉じるボタンがついているため使わない方は閉じることができます。
  • サブテキストはハッシュタグや後半の文章を入れることによりバリエーションをもたせます。
  • 大事なことを何度も伝えるときにサブテキストで言い換えると便利です。
  • また、いろいろなハッシュタグを使うことにより、いろいろな層にアプローチできます。
  • サブテキストはそれだけではなく、重複投稿をさける役割を担っています。
  • 重複投稿を避けるため、ランダムではなく、サブテキスト1 → サブテキスト2 → サブテキスト3の順番に動きます。

● 文字数1

  • 文字数は「メインテキスト + サブテキスト1」の合計値になっています。
  • サブテキストを使う場合は、この文字数があふれないようにしてください。

● サブテキスト2

  • サブテキスト1と一緒です。バリエーションBです。

● 文字数2

  • 文字数は「メインテキスト + サブテキスト2」の合計値になっています。
  • サブテキストを使う場合は、この文字数があふれないようにしてください。

● サブテキスト3

  • サブテキスト1と一緒です。バリエーションCです。

● 文字数3

  • 文字数は「メインテキスト + サブテキスト3」の合計値になっています。
  • サブテキストを使う場合は、この文字数があふれないようにしてください。

● 画像/動画url

詳細は、「Craft Twixの使い方(画像・動画管理編)」をみてください。

● ツリー投稿(プログラム用)

  • プログラミングから書き込んでいるため、原則いじらないでください。
  • 前のIDを覚えておく記憶用です。ここを消すとツリー投稿が動きません。ただし、投稿するたびに保存しているため、消えてもまた直りますので過度に心配しないでください。

● サブテキストカウンター

  • プログラミングから書き込んでいるため、原則いじらないでください。ただ中身を消す分には問題は起きません。
  • サブテキストがどのパターンを投稿しているのかをカウントして監視しています。

置き換えタグ

メインテキストとサブテキストに使える機能です。

必須の機能ではありませんが、乱数を活用した遊び心をサポートしています。
置換タグを利用して、ランダムな文字列の投稿が可能となります。
用途はアイデア次第で、例えばおみくじのような使い方も考えられます。

乱数は長い間、ボットやゲームアプリの開発でスパイスとして活用されてきました。簡単なものですが、データ作成にこだわりを持つ方や新しいアイデアを持っている方は、この機能をぜひ活用してみてください。

また、重複投稿を避ける手段としても活用でき、より安全な運用を実現できます。

文法

{random:文字列1,文字列2,文字列3}

複数の置換タグを指定することもできます。

メインテキスト

今日のごはんは{random:焼き鳥,牛丼,ハンバーグ}にします。
お野菜は{random:トマトサラダ,わかめサラダ,キャベツだけ}。

サブテキストでも置換タグを利用できるので、わけて使用することができます。

サブテキスト

デザートは{random:プリン,杏仁豆腐,ヨーグルト}。

ハッシュタグにも使えます。

日付の一括変換

「日付の一括変換」機能はスケジュール投稿の日付を一括でずらせます。
開始の年月日だけではなく、開始の時間や終了日にも対応するようアップデートしました。

使い方の例：

  • 新製品の発売日やキャンペーンがスケジュール変更になったとき、日付を一括でずらすことができます。
  • 再投稿したい場合にも便利です。
  • 終了日をずらしたい場合にも便利です。

仕様の説明です。

  • 日付や時間の列を一括置換します。
  • マイナス(-)とプラス(+)で時間を戻したり進めたりできます。
  • 日付、時間、終了日に対応しています。
  • 繰り上げ処理は行われます。(例)23時に2時間を加えると1時になり、日付も繰り上がります。
  • 未入力の場合は変更がないため、無視されます。
  • なるべく高速に動くようにしていますが、データ量が多いと少し置換に時間がかかる場合があります。お待ちください。
  • 編集 > 元に戻すで元の状態に戻せるため安心してください。
  • 編集 > やり直しで元に戻したものを再度進めることもできます。

動画のデモです。

繰り上げ処理を複雑に感じる方は、一つの項目のみを変更すると仕上がりがイメージしやすくなります。

認証が通りません!

認証はみなさん通過しているため、Twitter側で何か変更がない限り、1人だけ通らないことは考えにくいです。落ち着いて対応すれば、問題は解決できるでしょう。

APIキーとCallback URLsの指定間違いが最も多いケースです。

すべての項目で、1文字でも間違えていたらダメです。

半角スペースはプログラム上で自動的に削除されますが、コピーする際に1文字だけ選択し忘れたり、うっかり1文字だけ消してしまったりする可能性も考えられます。

APIキーはGAS（Google Apps Script）上からも確認できます。この方法も追加でお伝えしますが、開発時の確認方法ですので、誤操作にはご注意ください。ただ、確認の目的で見るだけなら問題はありません。

拡張機能 > Apps Script > 画面左の[歯車のアイコン(プロジェクト設定)] > スクリプト プロパティ

このスクリプトプロパティは、最終的にAPIキーなどの情報が保存される場所として使われます。そのため、キーを確実に確認したい場合やトラブルシューティングの際に、この確認手順を覚えておくと今後役立ちます。

直接、書き換えることはも可能ですが、他のキーを誤って削除するリスクがあるため、慣れていない方は避けた方がよいでしょう。万が一、おかしくなってしまってもし何か問題が発生した場合、スプレッドシートを再度コピーしてください。安全性の高い方法は、メニューから情報を再入力し、確認だけをこの手順で行う方法です。

また、もしCallback URLが置換前のURLのまま入力されていたり、置換時にミスがあったり1文字でも欠けていたりする場合、認証は成功しません。

重要な点として「Callback URLs」を更新した際は、アクセストークンも再取得が必要です。その後、アクセストクーンをメニューから再度入力することを忘れないでください。そのため、アクセストークンの入力用のポップアップとTwitter APIの入力欄はメニューで分けられています。

さらに、外部の要因で認証がうまくいかない場合も考えられます。

一時的なネットワークの不調
Twitterサーバとの通信が一時的に不安定となる場合、認証がうまく行かないことがあります。しばらく時間を置いてから、再試行してみてください。

ブラウザに関連した問題
異なるブラウザやプライベートブラウジングモードでの試行をおすすめします。

サードパーティの拡張機能やプラグイン、セッションやクッキーが問題の原因となることがあります。キャッシュのクリアが必要な場合もあります。

クッキーの削除も一つの方法ですが、他のサイトのデータをうっかりと削除してしまうリスクがあるため、新しいブラウザや別の異なるPCでの試行を推奨します。Chrome、Firefox、Edge、Safariなどのブラウザが利用できます。異なるブラウザを使っても、Googleアカウントでログインすれば、スプレッドシートの使用は容易です。

ファイアウォールやセキュリティソフト
一時的に無効化してみましょう。

ポストされません!

最もよくあるのは、日付などの必須項目が入力されていないことです。

特に忘れがちな設定は次のものです。

「作成中」→「完成」

この機能は、スプレッドシート上で執筆中の内容を安心して放置できるようにつけたものですが、すでに完成した原稿のみを持っている方はデフォルトを全部「完成」にしておくとこの制限を回避できます。

日にちの勘違いや、カレンダーで日付を指定する際、1年ずれているなどのミスにご注意ください。

すでに実施日が設定されている場合、日付入力を変更しても、実施日が優先されます。インターバル機能を使用すると、このような状況が発生しやすくなります。その場合、日付を指定した後にスケジュールを一時停止し、再度実行してください。

終了日の消し忘れもあるかもしれません。昔、設定した終了日を削除せず、それよりも未来の日付を設定すると、投稿は行われません。不要な終了日は全部消してください。

X(Twitter)側の一時的な通信不具合の可能性も考えられます。動画などの処理が複雑な操作を行う際、このような問題が発生する可能性が高まります。APIの利用に伴う特性を理解し、配慮してもらえると助かります。一部、どうしてもこちら側では改善が難しいも存在しますが、処理に関する部分もできる限り改善して参ります。

前提として、画像や動画は容量、FPS、尺などの制限内に収まる必要があります。

  • 画像: 最大5MB
  • 動画: 45MB弱、2分32秒以内、60fps

実施日が1970/01/01と表示される

実施日が1970/01/01と表示されるこのような現象は、不正確な記述が原因である可能性が考えられます。たとえば、時間を示すコロンがセミコロンになっている場合、このような症状が発生します。

17;00 → 17:00

正しい記述に修正してください。

スケジュールを追加する際、一度スケジュールを停止する必要は？

新しい行にスケジュールを追加する場合、停止する必要はありません。

しかし、既存の行を編集する際、一部の項目については停止する必要が生じることがあります。

もっと厳密に答えると、項目によって違うというのが答えです。

実施日はスケジュール実行開始時間とポスト後に設定されます。一度設定されたら変わりませんので、実施日を変えたいときは一度止める必要があります。

ただし、実施日以外の情報を変更する場合は、スクリプトを停止する必要はほとんどありません。仕組みとしては、1時間ごとにスプレッドシートをチェックし、60分以内の投稿を登録する形になっています。よって、この確認のタイミングに間に合えば、投稿は予定通りに実行されます。

もし1時間以内に投稿を予定している場合、次の確認まで待つ時間があるので、スケジュールを一度停止してください。スケジュールを再開すると、即時に確認が行われる仕組みになっています。

スクリプトがテキストなど投稿内容を取得するのは、投稿直前です。テキストの誤りを後から修正したり、ツリー投稿や画像投稿などの設定を後から追加したりすることは、投稿直前までに行えば反映されます。これらの項目を変更する際、スクリプトを停止する必要はありません。

上記のルールに基づいて使いこなせる方は使いこなしてください。

一応詳細を解説させてもらいましたが、ちょっと複雑だと感じる方は、必ず一度止めてクリアしたのち、スケジュールを再実行すると覚えておくだけでよいです。この運用方法でも一瞬でできる操作なのでそれでもいいのではないでしょうか。

URLや改行は何文字？

X(Twitter)のURLは1つにつき、11.5文字でカウントされます。改行は0.5文字です。

文字数がオーバーする場合は、ツリー投稿にして2つにわける方法がひとつの対策。
文末に「…続く」「以下補足」などを入れれば見てもらえる確率は増える気もします。

または、削れそうなところを考えて140文字に収めてください。多少の文字数オーバーは意外と創意工夫で収まります。

エラー Invalid or expired token

エラー

{"errors":[{"message":"Invalid or expired token","code":89}]}

このエラーは、X(Twitter)APIを使用すると、わりとよくでるエラーです。ちょっと運が悪かった系のエラーです…。スクリプトやGoogle Apps Script（GAS）の問題ではなくX(Twitter)側のエラーです。

表示されたエラーメッセージ「Invalid or expired token」は、訳すと「無効または期限切れのトークン」という意味です。つまり、何らかの理由でAPIが使えない状況にあるということです。

原因としては、次のような可能性が考えられます。

  • 古いAPIを流用している
  • 誤ってAPIキーを削除するなどの誤操作をしてしまった
  • Twitter側の更新やメンテナンスなどの理由で、タイミングが悪く新しく作成されたAPIトークンでも、すぐに期限切れになってしまう場合があります。

この問題の解決方法は、お手数をおかけして申し訳ないのですが、APIを新規作成して認証し直し、再度テストするしかありません。個人的にも最後のパターンのエラーを何度か経験しているため、お気持ちはわかります…。

すべてのAPIではなく、アクセストークンの再生成だけで問題が解決する場合もあります。そのため、まずはアクセストークンを再生成して、再度認証してください。

それでもダメな場合は、他のAPIキーを作り直すという手順が良いです。

スプレッドシートのアップデート

申し訳ございませんが、スプレッドシートという性質上、新しいバージョンのスプレッドシートがほしい場合は、再度一連の作業が必要です。こまめにアップデートする必要はないかもしれませんので、購入サイトにある更新履歴などを参考に必要と思うタイミングでアップデートしてください。

APIキーとコールバックURLを入れ替えれば、新しいスプレッドシートでも使用できます。

なお、コールバックURLは各スプレッドシートごとに異なります。コールバックURLを更新したら、アクセストークンも更新する必要がありますので、その点だけはご注意ください。X(Twitter) APIの認証も必要となります。上書きされるため、前のアプリを消す必要はありません。

他のAPIキーはそのまま使いまわせます。慣れれば、短時間で設定が完了できるようになります。

マニュアルのアップデート

マニュアルも改定を重ねています。新しいマニュアルがほしい方はお買い求めのサイトから再度ダウンロードしてください。

BOOTHの再ダウンロード手順は次のとおりです。

購入履歴 > 注文詳細画面へ > 商品名の右下に[ダウンロード]
